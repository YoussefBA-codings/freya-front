# freya-front (Freya Hub)

Outil interne de comptabilité (factures, commandes B2B, historique client). Vite + React 18 + react-router-dom + MUI 6. Backend : `APIs/shopify-back` (port 3000, `api.freya-hub.fr` en public — ne jamais couper cet accès, un webhook Shopify s'en sert).

## Déploiement

- Build : `dist/` servi en statique par nginx sur le serveur de prod (13.39.45.220), sous `/compta` (topologie Tailscale unifiée avec freyaOMS et Freya Portal — voir `tools/freyaOMS/docs/ARCHITECTURE.md`). `vite.config.ts` (`base: "/compta/"`) et `main.tsx` (`basename="/compta"`) doivent rester synchronisés.
- **Le serveur de prod n'a que 1,9 Go de RAM totale** (5 process PM2 + Postgres + nginx tournent déjà en permanence) — deux crashes serveur constatés le 2026-07-18 en buildant directement dessus, le second MÊME avec `NODE_OPTIONS=--max_old_space_size=2048` (le cap limite le tas d'UN process Node, pas la mémoire totale du système : si le reste tourne déjà à ~1,1 Go, il ne reste simplement pas assez de marge pour un build Vite lourd, peu importe le cap). **Règle : builder en LOCAL (`npm run build`) puis envoyer `dist/` sur le serveur par `rsync`/`scp`, jamais buildé directement sur le serveur.** Si un build sur le serveur est vraiment inévitable, capé au minimum avec `NODE_OPTIONS=--max_old_space_size=768 npm run build`, et seulement après avoir vérifié la RAM libre (`free -h`) — le serveur a 4 Go de swap (`/swapfile` + `/swapfile2`, ajouté le 2026-07-18 comme filet de sécurité) mais ça ralentit fortement, ce n'est pas une solution en soi.
- Authentification : la vraie barrière est désormais nginx (`auth_request` contre la session du portail Freya, cookie partagé) — `PrivateRoute.tsx` ne fait plus de vérification côté client (l'ancien flag `localStorage` était trivialement contournable). `Login.tsx` reste en place mais n'est plus atteint dans le flux normal (on se connecte une seule fois, sur le portail).
- **`.env.production` est OBLIGATOIRE avant tout build destiné au serveur.** Bug réel constaté le 2026-07-18 : builder en local avec seulement `.env` (valeurs de dev, `VITE_API_URL=http://localhost:3000/`) produit un `dist/` qui pointe vers `localhost:3000` une fois déployé - toutes les données restent muettes en prod, aucune erreur visible autrement qu'en ouvrant la console navigateur. Vite charge `.env.production` en priorité pour `npm run build` (mode production) - ce fichier contient la vraie `VITE_API_URL` (`https://api.freya-hub.fr/`) et N'EST PAS dans `.gitignore` (volontairement, comme la convention Vite le prévoit - il ne contient rien de secret, juste la config de prod). Après tout build, vérifier : `grep -o "api.freya-hub.fr\|localhost:3000" dist/assets/*.js`.

## Design — iso à 100% avec freyaOMS

Le thème (`src/theme.ts`) et la structure du shell (`src/elements/Nav.tsx`) sont volontairement **une copie exacte, pas juste similaire**, de `tools/freyaOMS` - décision équipe du 2026-07-18, "copier-coller", "on dirait le même projet", jusqu'au petit détail (taille de police d'un titre, couleur d'une bordure). Concrètement :

- `theme.ts` ne définit QUE ce que `tools/freyaOMS/src/lib/theme/theme.ts` définit lui-même (mêmes clés, mêmes valeurs : palette primary/secondary/success/warning/error/background/text, `shape.borderRadius: 12`, `typography.fontFamily` + `h4`/`h6` uniquement, `MuiAppBar`/`MuiCard`/`MuiButton`). Tout le reste (h1-h3/h5/subtitle/body, `MuiDataGrid`/`MuiChip`/`MuiTooltip`/`MuiTableCell`/`MuiOutlinedInput`, `divider`/`info`, l'échelle d'ombres) retombe sur les défauts MUI - EXACTEMENT comme freyaOMS, qui ne les surcharge pas non plus. Un premier passage avait gardé ces surcharges "en trop" (jugées comme de la polish sans opinion contraire côté freyaOMS) - corrigé le 2026-07-18 après retour utilisateur : ça produisait des tailles de titre différentes malgré une palette identique. Seule exception assumée : les imports de locale française (`coreFrFR`/`dataGridFrFR`/`pickersFrFR`) - question i18n, pas de design, freyaOMS n'en a simplement jamais eu besoin.
- `Nav.tsx` : boutons de nav copiés à l'identique (même `sx`, pastille verte pleine à la sélection, pas de `Tooltip` par item, pas de `<ListItem>` enveloppant), même largeur de drawer (248px), même breakpoint mobile (`md`), lien "Portail Freya" dans le drawer (pas l'AppBar), logout réel dans l'AppBar (voir `src/lib/portalAuth.ts` - appelle le `signOut` NextAuth du portail en `fetch` same-origin, puisque Hub et le portail partagent host+port, seul le chemin diffère ; déconnecte aussi freyaOMS, cookie de session partagé).

Pas de package partagé entre les deux repos : toute évolution du thème/shell côté freyaOMS (`src/lib/theme/theme.ts`, `src/components/DashboardNav.tsx`, `src/components/DashboardShell.tsx`) doit être reportée ici à la main pour ne pas diverger.

## Certificats SSL (freya-hub.fr / api.freya-hub.fr)

**Bug réel corrigé le 2026-07-18** : `api.freya-hub.fr` utilisait l'authenticator certbot `standalone` (a besoin de son propre port 80 libre) alors que nginx tourne en permanence dessus - tout renouvellement réel aurait échoué en boucle silencieusement. Corrigé en ré-émettant le certificat avec l'authenticator `nginx` (`certbot certonly --nginx -d api.freya-hub.fr --cert-name api.freya-hub.fr --force-renewal`), qui modifie la config nginx déjà en cours d'exécution au lieu d'avoir besoin du port pour lui-même - même mécanisme que `freya-hub.fr`, `catalogue.freya.tn` et `simulateur-locatif.com`, qui l'utilisaient déjà. Les 4 certificats du serveur partagent maintenant le même authenticator.

Deux hooks globaux obsolètes (`/etc/letsencrypt/renewal-hooks/{pre,post}/{stop,start}-nginx.sh`, qui arrêtaient nginx en entier avant CHAQUE renouvellement - hérités de l'ancien authenticator `standalone`) ont été supprimés : ils provoquaient un vrai conflit de bind (nginx redémarré deux fois, une fois par le hook et une fois par le plugin `nginx` lui-même) qui laissait le service dans un état "failed" côté systemd tout en continuant de servir du trafic (processus non suivi par systemd - `/run/nginx.pid` vide). Validé avec `certbot renew --dry-run` : les 4 certificats renouvellent proprement, sans hook, sans jamais couper nginx.
